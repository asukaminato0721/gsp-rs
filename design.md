# GSP 计算、几何与 IR 设计

## 目标

本项目把 `.gsp` 文档编译成一个可独立运行的 HTML 文件。最终文件同时包含：

- 文档的结构化数据；
- 几何与计算运行时；
- 交互和绘制代码。

编译结果是原始几何构造的可执行表示。点被拖动或参数发生变化后，所有依赖对象都应按照原有构造关系重新计算。

## 基本原则

### 载荷决定语义

几何关系、计算关系和交互关系必须来自 `.gsp` 载荷以及配套 `.htm` 中已有的定义。

不能根据当前坐标、对象名称或视觉结果猜测构造关系。坐标只表示某一时刻的结果，不能代替对象的定义。

当现有信息不足以完整表达一个对象时，应明确保留“不完整”状态，而不是加入只适用于个别样例的特殊逻辑。

### 构造关系优先于绘制结果

系统关心的核心不是“画出什么”，而是“这个对象如何由其他对象得到”。

例如，一个中点应表示为“由两个端点计算中点”，而不是一个固定坐标；一个交点应保留参与相交的两个对象以及交点分支，而不是只保留交点位置。

### 语义尽量集中在 Rust

载荷解析、表达式、几何运算、依赖关系和对象求值尽量由 Rust 实现，并复用于本地编译和浏览器 Wasm 运行时。

TypeScript 主要负责：

- 把 Rust 导出的数据绑定到浏览器；
- 处理鼠标、按钮等用户输入；
- 调用运行时重新计算；
- 把计算结果绘制为 SVG 和 HTML。

它不应发展出另一套独立的几何语义。

## 总体流程

```text
.gsp + 配套定义
        ↓
解析载荷与对象引用
        ↓
建立类型化场景 IR
        ↓
建立对象计算图和更新依赖图
        ↓
导出 JSON
        ↓
与 Rust/Wasm 运行时、TypeScript 查看器一起打包
        ↓
单个 HTML 文件
```

解析阶段只负责恢复原始文档表达的事实。场景 IR 负责把这些事实整理成稳定的内部模型。运行时图则把模型变成可重复执行的计算过程。

## 场景 IR

场景 IR 是载荷格式与浏览器运行时之间的边界。

它同时保存两类信息：

1. 对象当前的结果，用于初始显示；
2. 对象的构造定义，用于后续重新计算。

因此，一个几何对象通常既有初始几何数据，也有可选的计算定义。前者是快照，后者才是对象长期有效的语义。

场景中的主要概念包括：

- 几何对象：点、线、圆、圆弧、多边形等；
- 计算对象：参数、标量、表达式、函数和测量值；
- 交互对象：按钮、可拖动控制点和动态标签；
- 重复构造：迭代及其状态；
- 显示属性：颜色、线型、可见性和文本样式；
- 来源信息：对象对应的原始载荷位置和引用关系。

场景 IR 使用明确的对象引用连接这些内容，而不是在运行时重新识别它们之间的关系。

## 对象计算图

计算模型的主体是一个有向无环图：

```text
对象 = 操作(父对象...)
```

图中的节点分为两类：

- 源节点：可以由用户或外部状态直接改变，例如自由点和参数；
- 派生节点：由一个操作和若干父节点计算得到。

父节点顺序是操作语义的一部分。例如线段的起点和终点、角的顶点和两边、交点的左右输入都必须保持明确角色。

求值前需要验证：

- 节点标识不重复；
- 所有父节点存在；
- 图中不存在环；
- 操作收到的父值数量和类型正确。

验证通过后，节点按拓扑顺序计算。这样每个对象求值时，它依赖的对象已经得到最新结果。

## 统一的值与操作模型

对象图中的值不仅是坐标，还包括标量、点、线、圆、圆弧、点集、多边形集、颜色和文本等。

操作表负责定义这些值之间允许发生的计算。操作大致包括：

- 基本几何构造；
- 投影、相交和相切；
- 平移、旋转、缩放和反射；
- 长度、角度、面积和路径参数等测量；
- 表达式与函数采样；
- 轨迹和迭代；
- 动态颜色。

所有操作通过统一接口求值，而不是让每种场景对象各自维护一套更新流程。这样新增一种构造时，主要工作是补充载荷到操作的映射，以及对应的 Rust 操作实现。

## 统一 enum 设计

几何值可以由一个统一的 enum 表示。概念上可以写成：

```rust
enum ObjectValue {
    Scalar(f64),
    Geometry(GeometryValue),
    Color(Color),
    Text(String),
    GeometryList(Vec<GeometryValue>),
}

enum GeometryValue {
    Point(Point),
    Line(LineValue),
    Circle(CircleValue),
    Arc(ArcValue),
    Polygon(Vec<Point>),
    Curve(CurveValue),
}

struct Point {
    x: f64,
    y: f64,
}

struct LineValue {
    kind: LineKind,
    start: Point,
    end: Point,
}

enum LineKind {
    Segment,
    Line,
    Ray,
}
```

这里的 `ObjectValue` 表示对象在某一时刻计算出的值。线段、直线和射线具有相同的数据形状，只通过 `LineKind` 区分语义，因此不需要分别维护三套坐标结构。

几何构造和几何变换也由统一的操作 enum 表示：

```rust
enum Op {
    Construct(ConstructOp),
    Transform(TransformOp),
    Intersect(IntersectOp),
    Measure(MeasureOp),
    Evaluate(EvaluateOp),
    Curve(CurveOp),
    Iterate(IterateOp),
}

enum TransformOp {
    Translate(Translation),
    Rotate(Rotation),
    Scale(Scale),
    Reflect(Reflection),
}

enum Translation {
    ByVector,
    ByPointPair,
    ByScalars,
    ByDelta { dx: f64, dy: f64 },
}

enum CurveOp {
    FunctionPlot,
    ParametricCurve,
    CoordinateTrace,
    PointTrace,
    CustomTransformTrace,
}
```

不同平移形式表示不同的输入语义。例如固定 `dx/dy` 不需要额外父对象，向量平移需要一个向量父对象，而点对平移需要两个点来定义位移。

操作本身不直接递归包含完整父对象。父子关系统一保存在 DAG 节点中：

```rust
enum ObjectDefinition {
    Source {
        initial: ObjectValue,
    },
    Derived {
        op: Op,
        parents: Vec<ObjectId>,
    },
}

struct ObjectNode {
    id: ObjectId,
    definition: ObjectDefinition,
}
```

例如线段应表示为：

```text
segment = Construct(Line(Segment))(point_a, point_b)
```

而不是把 `point_a` 和 `point_b` 的完整对象递归复制到线段定义中。使用对象标识引用父节点，才能保持对象身份、共享依赖并支持增量更新。

### 统一表示的边界

“对象统一表示”应指计算值和计算操作统一，而不是把所有职责塞进同一个 enum。

以下内容应与 `ObjectValue` 分开：

- 对象标识和父子依赖；
- 颜色、线宽、可见性等绘制属性；
- 原始载荷来源；
- 可拖动性和按钮关系；
- 约束参数以及拖动时的反向映射规则。

因此可以把系统理解成三层：

```text
ObjectNode       定义对象如何计算
ObjectValue      保存本次计算结果
RenderEntity     说明如何显示和交互，并引用 ObjectNode
```

这种拆分既能获得统一 enum 的好处，又不会让几何值类型承担渲染、交互和来源追踪等无关职责。

### 与当前实现的比较

当前代码已经实现了这个设计的大部分核心：

- 运行时已有统一的对象值 enum；
- 构造和计算已有统一的操作 enum；
- 函数曲线、参数曲线和各类轨迹统一产生曲线值，并由曲线操作 enum 保留生成语义；
- 自定义变换点和自定义变换轨迹共享同一变换程序；
- 点、线、圆、弧和多边形共用一套几何变换描述；
- 对象节点使用 `操作 + 父节点` 形成 DAG；
- 类型化对象图由场景 IR 直接持有；
- 源节点显式说明它绑定到参数、几何对象还是约束控制量；
- Rust 操作表统一执行这些操作。

载荷解码仍可按点、线、圆等类型分别组织，因为不同载荷结构本来就需要不同的读取规则；但这种分类只存在于提取器内部。提取阶段结束前必须把有效构造统一转换成对象图，HTML 导出层只负责序列化，不再二次推导几何关系。

对象图完整性也是编译契约。若载荷中的父对象或操作尚不能进入统一图，编译器会报告具体未完成节点并停止导出，不再把静态坐标连同一套浏览器兼容算法交给页面。浏览器中的派生几何、曲线、轨迹和几何迭代统一由 Rust/Wasm 对象图求值；TypeScript 只同步输入、应用结果以及更新标签和表格等显示状态。

因此，当前边界可以概括为“载荷类型负责解码，对象图负责全部正向计算，渲染实体只消费结果”。约束和变换绑定仍可作为拖动时的反向映射信息，但不能成为另一条正向求值路径。

### 实现边界

统一模型不等于用一个巨大的 enum 替换所有场景数据。对象图是唯一计算 IR；绘制对象、样式和交互信息仍保持独立。载荷提取器可以使用临时的类型化绑定帮助解码，但这些绑定不能越过对象图形成第二套正向计算语义。

新增构造时，需要同时满足三个条件：能由载荷引用生成对象节点，能由 Rust 操作表求值，能由现有结果绑定完成绘制。缺少任一环节都应在编译期报告，而不是恢复浏览器兼容分支。

最终约束不是“所有数据只有一个 enum”，而是“所有计算只有一个语义入口”。

## 曲线与轨迹

函数曲线、参数曲线和轨迹都表示一组有顺序的几何点，但它们的定义来源不同。IR 应把它们统一为曲线值，同时保留各自的生成操作。

```text
CurveValue = CurveOp(父对象、定义域、采样策略)
```

函数曲线由一个表达式生成；参数曲线由同一参数驱动两个坐标表达式；轨迹则在一系列驱动状态下重复执行已有对象图，记录目标对象的结果。它们最终都得到 `CurveValue`，因此相交、约束、绘制和变换可以消费同一种值，而不需要识别曲线最初来自哪种场景对象。

曲线定义与采样结果需要分开理解：

- 曲线操作保存表达式、驱动关系、定义域和采样规则；
- 曲线值保存本次求值得到的有序样本，并保留样本位置；
- 绘制层只消费曲线值，不重新解释表达式或轨迹；
- 交点和曲线上点等操作依赖曲线节点，而不是依赖某条查看器专用折线。

### 轨迹

轨迹不是一种新的几何计算语言。它由驱动源、目标对象和普通对象图程序组成：运行时改变驱动源，重新执行同一 DAG，再收集目标值。

点轨迹收集点，线段轨迹可以组合两条端点轨迹。坐标轨迹是表达式驱动的特例。这样轨迹中的中点、交点、变换和约束都复用普通构造操作，不在 TypeScript 中复制一套近似算法。

某些驱动状态下几何对象可能没有定义，例如平行线没有交点。对象图用统一的“未定义值”表达这种状态，并沿 DAG 传播；轨迹保留采样位置，只收集有定义的结果。线段轨迹按相同采样位置配对两个端点，不能把两侧独立丢点后的序列按下标硬拼。

### 自定义变换

自定义变换由距离表达式、角度表达式、参数映射和尺度约定共同定义。单点变换与自定义变换轨迹应共享同一个变换程序：

```text
CustomTransformPoint = CustomTransformProgram(源值)
CustomTransformTrace = sample(CustomTransformProgram, 驱动区间)
```

前者求一个驱动状态，后者在定义域中求多个状态。共享程序可以保证静态点与其轨迹使用完全相同的距离和角度语义。

### 函数曲线与参数曲线

普通函数曲线把自变量映射为平面点，参数曲线把同一参数分别送入横、纵坐标表达式。极坐标函数只是函数曲线的另一种坐标解释，不应形成独立运行时。

表达式引用的参数都是 DAG 父节点。参数变化时，曲线节点重新采样；浏览器只接收新的 `CurveValue` 并重绘。

## 几何约束与绑定

点的模型需要区分“约束”和“绑定”。

约束描述点可以在哪里，例如在线、圆、圆弧或多边形边界上。约束中保存可由用户改变的参数，例如在线上的位置或圆周方向。

绑定描述点或其他对象如何由父对象计算得到，例如中点、交点、变换结果或表达式坐标。

这一区分形成了清晰的交互过程：

```text
用户拖动
    ↓
修改源点或约束参数
    ↓
沿对象图重新计算派生对象
    ↓
更新绘制结果
```

拖动派生对象时，系统应尽量反求它允许修改的源状态，而不是把派生结果强行改成自由坐标。这样交互之后，原来的几何关系仍然成立。

## 计算模型

参数是可变输入，标量是测量或派生结果，表达式描述它们之间的计算关系。

表达式被解析成结构化语法树。表达式节点可以引用变量、常数、参数以及一元或二元运算。

函数、动态标签、轨迹和迭代都建立在同一个表达式模型上。它们的区别主要在于输入方式和输出形式，而不是各自拥有不同的计算语言。

测量值也应进入对象图。长度、角度、面积等结果既可以显示在标签中，也可以继续作为其他表达式或几何构造的输入。

## 迭代

迭代也是对同一构造过程的重复执行，但它会把上一轮结果作为下一轮状态。

它与轨迹都复用普通对象图中的操作，避免产生一套只用于采样的近似逻辑。区别在于轨迹的各次采样由驱动值独立决定，而迭代存在显式的状态传递。

这种表示能够保证：动态场景中的对象如何运动，轨迹就如何生成；普通构造如何计算，迭代步骤就如何计算。

## 更新依赖图

对象计算图回答“对象的值如何得到”，更新依赖图回答“某个输入变化后需要执行哪些更新阶段”。两者用途不同。

当参数或源对象发生变化时，运行时从对应的脏节点出发，找出受影响的对象，并按依赖顺序执行必要的计算、迭代和标签更新。

依赖关系由 Rust 根据场景 IR 生成，浏览器只负责提交变化和执行计划。这样可以避免 TypeScript 通过遍历场景并猜测依赖关系。

## 不完整构造的处理

导出对象图时，如果某种现有场景定义还不能转换成统一操作，应记录未完成项，并把整个几何图标记为不完整。

不完整对象图不能进入 HTML 导出。初始坐标可以用于提取诊断，但不能冒充派生定义，也不能触发浏览器兼容计算。新增语义必须先进入 Rust 操作表和对象图，页面才可以消费它。

## 数据导出与运行时

Rust 结构体是 JSON 格式的定义来源，并生成 TypeScript 类型。导出的 JSON 包含场景对象、计算定义、交互信息和依赖图。

最终 HTML 内嵌 JSON、Wasm、查看器代码和样式，因此不依赖外部服务即可运行。

浏览器初始化时读取场景快照；发生交互后，把最新源值交给 Wasm 对象图求值，再将结果写回运行时场景并重绘。

## 设计约束

- 原始载荷引用是构造关系的来源。
- 当前坐标是结果，不是定义。
- 对象之间通过明确引用连接，不通过名称或几何相似性匹配。
- 用户修改源状态，派生状态通过计算获得。
- 表达式、几何操作和依赖分析尽量只有一份 Rust 实现。
- TypeScript 负责绑定和显示，不承担载荷语义恢复。
- 不支持的语义必须可见，不能被静默近似。
- 新增支持时应同时补充载荷解析、IR 映射、运行时操作和测试。
